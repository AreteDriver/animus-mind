# Migration Guide: Animus v2.3 Assistant → Animus-Mind v2.1+

**Date:** 2026-06-30  
**Scope:** Port working components from the stable Assistant codebase into the Mind-class architecture  
**Rationale:** ADL-20260618-001 rejected the hybrid middle ground. The correct path is preserve + fork.

---

## Executive Summary

This guide describes how to migrate from `animus` (Assistant-class, v2.3 stable) to `animus-mind` (Mind-class, v2.1+ architecture). It is **not** an in-place upgrade. The source repository remains frozen as `v2.3-assistant-stable`. The target repository is a clean scaffold that selectively copies proven components while adopting the v2.1 structural design.

**Core principle:** Reuse what works. Redesign what doesn't. Archive what is dead.

## 3. What Stays Behind (Do Not Copy)

### 3.1 Dead Blockchain Packages (`packages/_archive/`)

**Action:** Do not copy. Archive externally if needed.

- `animus-validator` — Staking, node rewards
- `animus-prospector` — Crypto opportunity hunter
- `animus-faucet` — Faucet farming
- `animus-content` — Write-to-earn publishing
- `animus-bounty` — Task claiming
- `animus-mev` — MEV extraction
- `animus-arbitrage` — DEX arbitrage
- `animus-referral` — Referral stacking

**Rationale:** Mind-class architecture has no crypto scope. These packages are not merely unmaintained — they are **out of domain**.

### 3.2 SQLite/JSON as Canonical Persistence

**Action:** Do not copy `LocalMemoryStore` or `JsonMemoryStore` as canonical stores.

**Rationale:** Mind architecture requires PostgreSQL as the durable authority. SQLite/JSON may exist only as **rebuildable projections** (search index, vector cache, graph cache). The `MemoryLayer` auto-selection logic (`backend="auto"`) must be replaced with explicit canonical vs. projection distinction.

### 3.3 Package-Per-Function Layout

**Action:** Do not recreate `packages/core/`, `packages/forge/`, `packages/quorum/`, `packages/bootstrap/`.

**Rationale:** The current layout has unclear ownership boundaries and version divergence (0.1, 0.8, 1.2, 1.9, 2.3). Mind architecture uses **module-per-capability** with a single unified release version:

| Old Package | New Module | Version |
|---|---|---|
| `animus-core` (2.3.0) | `modules/object_core/` + `modules/memory_service/` | unified |
| `animus-forge` (1.9.0) | `modules/agent_runtime/` + `modules/trace_eval/` | unified |
| `convergentAI` (1.2.0) | `modules/agent_runtime/` (consciousness quorum subcomponent) | unified |
| `animus-bootstrap` (0.8.0) | `apps/api/` + `apps/cli/` | unified |
| `animus-kernel` (0.1.0) | `modules/object_core/` + `modules/projections/` | unified |
| `animus-types` (0.1.0) | `contracts/schemas/` + generated Python types | unified |
| `animus-contracts` (0.1.0) | `contracts/` (expanded to 22+ schemas) | unified |

### 3.4 Imperative API Call Patterns

**Action:** Do not copy direct function-call coupling between components.

**Rationale:** Current code crosses boundaries with Python dicts and direct imports. Mind architecture uses an **event bus + outbox pattern**. Services consume events, not call functions. This enables:
- Replay
- Traceability
- Idempotency
- Async scaling

## 7. Risk & Mitigation

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Over-copying legacy patterns | Medium | High | Code review every copied file against v2.2 engineering doctrine (Section 3 of roadmap) |
| Under-copying working code | Medium | Medium | Run smoke tests after each module port; if it breaks, re-copy |
| Schema drift during port | Medium | High | Schema compiler runs in CI; contract compatibility check blocks breaking changes |
| PostgreSQL complexity for personal deploy | Low | Medium | Keep Docker Compose local setup; SQLite as projection rebuild target only |
| Scope creep back to general product | Medium | High | Charter scope boundary; personal-use mandate in README |
| v2.3 bug fixes needed while building Mind | Low | Low | v2.3 is frozen; critical fixes cherry-pick to both repos |
